# Architecture

**Analysis Date:** 2026-03-28

## Pattern Overview

**Overall:** Layered React SPA with Zustand state management, event-driven progression system, and decoupled game mechanics engine.

**Key Characteristics:**
- Vite + React 19 + TypeScript for type-safe UI development
- Zustand with persistence middleware for centralized game state
- Event-driven architecture with pub/sub event bus for decoupled systems
- Separate concerns: UI (React), State (Zustand slices), Engine (pure functions), Data (static game configs)
- Appwrite backend integration for user accounts and persistence (optional)
- 200ms tick cycle driving all gameplay state updates

## Layers

**UI/Presentation:**
- Purpose: React components for user interaction and visual feedback
- Location: `src/ui/`
- Contains: Panel components (FarmPanel, MachinePanel, etc.), Layout wrapper, Sidebar navigation, StatsBar
- Depends on: Zustand game store, event bus for notifications
- Used by: React DOM renderer

**State Management:**
- Purpose: Single source of truth for all game state using Zustand
- Location: `src/store/`
- Contains: `gameStore.ts` (root store) and slice creators in `slices/`
- Depends on: Mechanics engine, services (eventBus, marketService)
- Used by: All UI components via `useGameStore()` hook

**Store Slices:**
- `resourceSlice.ts`: Coins, gems, lifetime currencies
- `farmSlice.ts`: Plots, crop planting/harvesting
- `automationSlice.ts`: Machines, workers, upgrades
- `metaSlice.ts`: Unlocked content, skill points, panel state, prestige
- `storySlice.ts`: Chapter progress, boss HP, quests, weapons
- `workerAssignmentSlice.ts`: Worker deployment and bonuses
- `craftingSlice.ts`: Recipe queue and crafting state
- `trackingSlice.ts`: Event tracking for unlock conditions (harvests, boss damage, region rep, crafting)

**Game Engine:**
- Purpose: Pure calculation functions for game rules and mechanics
- Location: `src/engine/`
- Contains:
  - `mechanics.ts`: Core calculations (crop growth, machine production, skill multipliers)
  - `productionEngine.ts`: Tunable production multiplier system with 3 knobs (workers, skills, prestige)
  - `TickEngine.tsx`: React component running 200ms tick cycle
  - `liveOps.ts`: Dynamic events and merchant offers
- Depends on: Data definitions (crops, machines, skills), types
- Used by: Game store during tick, UI for previews

**Services:**
- Purpose: External integrations and cross-cutting concerns
- Location: `src/services/`
- Contains:
  - `appwrite.ts`: Backend client configuration and connectivity checks
  - `eventBus.ts`: Global pub/sub system for game events + React hooks
  - `marketService.ts`: Dynamic pricing with tunable parameters
  - `unlockService.ts`: Unlock condition evaluation engine
- Depends on: None (except eventBus by others)
- Used by: Store, engine, UI components

**Data Layer:**
- Purpose: Static game configuration and content definitions
- Location: `src/data/`
- Contains: Crops, machines, skills, workers, chapters, recipes, narrative, world regions
- Files: All auto-generated by economy generation scripts (crops.ts, machines.ts, etc.)
- Depends on: Type definitions
- Used by: Engine, services, UI for lookups

**Type Definitions:**
- Purpose: Shared TypeScript interfaces for type safety
- Location: `src/types/`
- Contains:
  - `base.ts`: Core game objects (Plot, PlayerMachine)
  - `game.ts`: Slice interfaces and GameState composition
  - `unlock.ts`: Unlock condition types
  - `worker.ts`: Worker bonuses and assignments
  - `regionChapter.ts`: Story progression types
- Depends on: None
- Used by: All layers for type safety

## Data Flow

**Tick Cycle (200ms):**

1. TickEngine component calls `store.tick(timestamp)`
2. Game store tick function:
   - Calculates crop growth for each plot (using mechanics.calculatePlotGrowth)
   - Calculates machine coin production (mechanics.calculateMachineProduction)
   - Updates market prices (if >60s since last update)
   - Emits COINS_CHANGED event if coins changed
   - Calls checkQuestProgress() and processCraftingQueue()
3. Event bus emits COINS_CHANGED, triggering unlock evaluation
4. Unlock pipeline checks if new content requirements are met
5. UI components re-render via Zustand subscriptions

**Player Action Flow:**

1. User clicks button (e.g., "Plant Crop")
2. UI calls store method (e.g., `plantCrop(plotId, cropId)`)
3. Store updates state slice
4. Event bus emits appropriate event (e.g., CROP_PLANTED)
5. Event subscribers (VillageNews, unlock pipeline) respond
6. UI re-renders via Zustand observer pattern

**Unlock & Progression Pipeline:**

1. Player progresses: completes quests, earns coins, defeats bosses, etc.
2. Events emitted: COINS_CHANGED, QUEST_COMPLETED, BOSS_DEFEATED, etc.
3. StorySlice.checkQuestProgress() evaluates unlock conditions
4. UnlockService.evaluateCondition() checks each condition against state
5. When all requirements met: CONTENT_AVAILABLE event emitted
6. UI shows notification, player can then accept/unlock
7. CONTENT_UNLOCKED event emitted, feature becomes active

**State Persistence:**

- Zustand persist middleware saves to localStorage under 'cozy-garden-save' key
- Auto-saves on every state change
- Loaded from localStorage on app startup
- Appwrite backend optional for cloud sync (connection checked but not required)

## Key Abstractions

**ProductionMultiplier:**
- Purpose: Combines worker bonuses, skill bonuses, prestige, and fixed multipliers into single value
- Location: `src/engine/productionEngine.ts`
- Pattern: Single equation with 3 tunable knobs (AURA_WEIGHT, SKILL_CAP, PRESTIGE_SCALE) for balancing
- Used by: `calculateMachineProduction()` to compute final coin generation

**EventBus (Pub/Sub):**
- Purpose: Decoupled event system for game events
- Location: `src/services/eventBus.ts`
- Pattern: Singleton with typed event handlers, wildcard subscriptions, event log
- Event types: 60+ typed events (CROP_PLANTED, WORKER_ASSIGNED, COINS_CHANGED, etc.)
- React hooks: `useGameEvent()`, `useRecentEvents()`, `useNewsEvents()`

**UnlockService:**
- Purpose: Evaluates unlock conditions independently from store
- Location: `src/services/unlockService.ts`
- Pattern: Pure functions taking UnlockCondition + UnlockEvaluationState, returning boolean
- Conditions: chapter_started, quest_completed, coins_earned, harvest_count, boss_damage, craft_recipes, etc.

**WorkerAssignmentSlice:**
- Purpose: Complex worker deployment system with bonuses
- Location: `src/store/slices/workerAssignmentSlice.ts`
- Pattern: Tracks worker instances, assignments (plot/machine), efficiency bonuses
- Used by: Production engine for global aura and machine-specific bonuses

**Tunable Systems:**
- Purpose: Parametrized balance knobs for testing without code changes
- Examples:
  - ProductionEngine: 6 tunables for worker/skill/prestige balance
  - MarketService: 3 tunables for price volatility/trend/weather
  - All have safe ranges defined for validation

## Entry Points

**Application Root:**
- Location: `src/main.tsx`
- Triggers: Browser loads HTML, loads React app
- Responsibilities: Initializes ReactDOM, pings Appwrite, renders App component

**App Component:**
- Location: `src/App.tsx`
- Triggers: Loaded by main.tsx
- Responsibilities: Sets up Layout, PanelManager, verifies Appwrite connectivity

**Layout Component:**
- Location: `src/ui/Layout.tsx`
- Triggers: Rendered by App
- Responsibilities: Renders TickEngine (invisible), StatsBar, Sidebar, main content area, VillageNews sidebar

**TickEngine:**
- Location: `src/engine/TickEngine.tsx`
- Triggers: Rendered on every Layout mount
- Responsibilities: Sets up 200ms interval timer, calls store.tick() with delta time

**index.html:**
- Location: Root directory
- Triggers: Browser loads page
- Responsibilities: Vite entry point, renders into #root div

## Error Handling

**Strategy:** Try/catch in event handlers, console logging, graceful degradation

**Patterns:**
- Event bus handlers wrapped in try/catch to prevent one handler crash from stopping others
- Appwrite ping at startup warns on failure but doesn't block app
- Unlock condition evaluation returns false on missing data rather than throwing
- Missing crop/machine/skill definitions treated as "not found" not errors

## Cross-Cutting Concerns

**Logging:** Browser console only. Pattern: `console.log()`, `console.error()` with context prefix.

**Validation:**
- TypeScript strict mode catches most issues at compile time
- Runtime validation in unlock conditions and production calculations
- Data validation during economy generation (economy/generate_*.ts)

**Authentication:**
- Appwrite optional client auth (no session management currently)
- localStorage persists save data per browser, no user accounts

**Styling:**
- Tailwind CSS for utility classes
- CSS variables for theme (--bg, --panel, --brown-border, etc.)
- Inline styles for dynamic values in React components

**Performance:**
- Zustand observer pattern ensures only subscribed components re-render
- useMemo() in PanelManager to prevent recreation on activePanel change
- Event handlers are memoized to prevent subscription churn
- Tick cycle batches updates in single state transaction
